# 0023: National all-routes map via PMTiles and MapLibre

Status: accepted (2026-06)

## Context

The national map (`/map/`, ADR-era geo work) places each agency as a single
point. The next question a reviewer asks is "show me the actual service" — every
agency's route lines on one map. The per-agency route extraction (route_geometry,
the stacked branch this builds on) already emits one deduplicated `LineString`
per route in each agency's `data/artifacts/<id>/geometry.geojson`. Aggregating
those across ~1,400 agencies is tens of thousands of lines and hundreds of
thousands of vertices: too much to ship as one GeoJSON the browser parses on
load, and the site has no tile server (it is static files on GitHub Pages).

## Decision

Render the national all-routes map from a single **PMTiles** archive of vector
tiles, read client-side by **MapLibre GL** (already pinned at 4.7.1 for the
existing maps) through the `pmtiles://` protocol.

PMTiles is a single-file, cloud-optimized tile archive: the client fetches only
the tiles in view via HTTP **range requests**, so a national archive costs only
the bytes on screen, not a full download. Static hosts that honor range requests
(GitHub Pages, S3) serve it with no tile server. deck.gl was considered and
rejected: it has no native PMTiles layer, and for line rendering at this scale
MapLibre's vector-tile path is the faster, static-hostable option.

Pipeline:

1. `national_routes.build_national_routes` gathers the `kind == "route"`
   LineStrings from every agency's geometry artifact (stops are excluded — a
   national point cloud of every stop is both unreadable and enormous), tagging
   each line with `agency`, `agency_name`, `route` (label), `type` (route-type
   family in words), and `grade` (from `catalog.json`). Output is deterministic
   newline-delimited GeoJSON.
2. `scripts/build_national_pmtiles.py` runs **tippecanoe** over that GeoJSONL to
   build zoom-aware, aggressively simplified vector tiles and writes them
   straight to `web/tiles/national-routes.pmtiles`.
3. `/routes/` (rendered by `render_site`) loads the archive, draws every route
   line, lets the reader recolor by route type or by agency grade (each with a
   text-labeled legend, never color alone), and on click pops up the agency,
   route, type, grade, and a link to the scorecard.

## Hosting: commit to `web/`, with a size threshold

GitHub Pages returns `accept-ranges: bytes` and serves `206 Partial Content`
(verified against the live host), so it satisfies PMTiles' only hosting
requirement with no extra infrastructure. We therefore **commit the archive into
`web/tiles/`** and let it deploy with the rest of the static site, **as long as
it stays at or under 25 MB** — comfortably under GitHub's 50 MB warning and
100 MB hard limit, and a non-issue for a phone over 4G because range requests
only pull visible tiles. If the archive ever exceeds 25 MB, move it to the
project's existing S3 + CloudFront (which also serves range requests and sets
the CORS the cross-origin range fetch needs) and point the page's `pmtiles://`
URL there instead; the build script prints the size so the decision is explicit
each run.

At the time of writing only the two pilot agencies carry geometry artifacts in
the repo, so the committed archive is ~6 KB (35 routes). At full national scale
the simplified line archive is expected to be a few MB — still well under the
threshold, so the commit-to-`web/` choice holds; revisit only if the size report
crosses 25 MB.

## tippecanoe is a build-time dependency, kept off the daily build

tippecanoe is not in the daily render image, so the tile build is deliberately
**not** part of `scorecard render-site` or the daily Pages deploy: those would
break. The aggregation and the `/routes/` page render with no tile tooling (the
page reads the already-committed archive; `render_site` only aggregates route
counts for the page copy). Regenerating the archive is a separate, on-demand step:
`make tiles` locally, or the dedicated, SHA-pinned `tiles.yml` workflow
(`workflow_dispatch`) that builds tippecanoe from a pinned source tag, rebuilds
the archive, and commits it back only when it changed. Because the archive is
regenerated by hand rather than on the daily cron, it never blocks the daily
deploy and never churns on a schedule. The newline-delimited GeoJSONL is
byte-reproducible; the
PMTiles bytes are **not** guaranteed reproducible (tippecanoe embeds build
metadata and may reorder features), so the archive is treated as a generated
asset, not a checked invariant.

## Accessibility: an enhancement, not the data interface

A national map of millions of route segments cannot be a literal accessible data
table, so we do not pretend the canvas is one. `/routes/` is documented in
`docs/vpat.md` as an **exploratory enhancement (Partially Supports)**, not an
AAA-conformant data path. The canvas is `aria-hidden`, carries no focusable
controls, is out of the tab order, and honors `prefers-reduced-motion`. The
conformant equivalents already exist and are linked at the top of the page behind
a "Skip to the accessible agency list" bypass: the sortable `/agencies/` list,
the `/leaderboard/`, and each scorecard's per-agency route table. The page passes
the axe/pa11y gate with zero violations.

## Consequences

- The national all-routes view ships with no new runtime infrastructure: one
  committed file, served by the existing static host over range requests.
- The tile build needs a human or the scheduled workflow to run tippecanoe; the
  daily build is untouched and cannot be broken by a missing tile tool.
- The committed archive can lag the latest geometry between tile builds; that is
  acceptable for a reference map and is the same staleness model as the rest of
  the published site between renders.
